# Modules

## Render modules

| ID | Category | Description | Settings | Limitation |
| --- | --- | --- | --- | --- |
| `fullbright` | Render | Locally increases brightness with reversible gamma or Night Vision. | `use_gamma`, `night_vision`, `brightness` | Gamma is constrained to Minecraft's 0.0–1.0 option range; Night Vision is visual and client-local. |
| `anti_blind` | Render | Hides selected local impairment visuals. | `blindness`, `darkness`, `nausea`, `pumpkin_overlay`, `powder_snow_overlay` | Does not remove effects from the local player or change server state; it only filters their 26.2 render paths. |
| `anti_debuff` | Render | Hides the local swirling particles emitted by negative potion effects. | `effect_particles`, `harmful_only` | It redirects only the local player's client-side effect-particle spawn in `LivingEntity.tickEffects`; effects, durations, screen overlays, and other entities are untouched. Minecraft renders one merged particle stream per entity, so `harmful_only` (default) hides particles only while every active effect is harmful and shows them whenever a beneficial effect is also active. It removes no effect and changes no server state. |
| `camera_distance` | Render | Sets a local desired third-person camera distance. | `distance` | Camera collision clipping still applies, so the visible distance can be shorter near blocks. First-person view, player position, packets, and server state are unchanged. |
| `camera_no_clip` | Render | Prevents local third-person camera collision pull-in. | none | It changes only the detached camera's local visual collision-distance calculation. The player, world blocks, first-person camera, server collision, and all packets are unchanged; the camera can visually pass through terrain. |
| `no_fire_overlay` | Render | Hides the local first-person fire overlay. | none | It suppresses only the 26.2 local `ScreenEffectRenderer` fire branch. Burning state, damage, sounds, particles, and server state remain unchanged. |
| `no_fog` | Render | Extends local fog planes to reduce visible fog. | `distance` | It raises only the local fog plane distances, bounded to 16–4096 blocks, and never makes a vanilla plane nearer. It does not load chunks, reveal unloaded terrain, change fluids/weather, or affect server visibility rules. |
| `no_hurtcam` | Render | Hides local camera shake caused by taking damage. | none | It replaces only the active hurt-animation value read by the local `GameRenderer`; health, damage, knockback, death-camera rotation, effects, and all server state remain unchanged. |
| `no_shield_overlay` | Render | Hides the raised local first-person shield overlay. | none | It changes only this client's first-person pose for an actively used vanilla shield; the shield remains visibly held, while blocking, damage reduction, timing, inventory state, and server processing are unchanged. |
| `anti_totem_animation` | Render | Hides the local death-protection item activation overlay. | none | It suppresses only the local `GameRenderer` visual activation for an item with Minecraft's death-protection component. Totem effects, particles, sound, inventory changes, and server processing remain unchanged. |
| `dinnerbone` | Render | Applies Minecraft's upside-down transform to selected local living-entity models. | `players`, `hostiles`, `passive` | It affects only the local `LivingEntityRenderer` result and preserves vanilla's existing Dinnerbone/Grumm behavior. Players and `Monster` subclasses are distinct selectable categories; other living entities use the passive category. It does not change pose, data, collision, or packets. |
| `rainbow_enchant` | Render | Tints the local enchantment glint emitted for item stacks. | `color`, `rainbow`, `rainbow_speed` | It wraps only the 26.2 `ItemFeatureRenderer` foil vertex color. The item, enchantments, textures, server state, and worn armor-layer glint remain unchanged. |
| `better_crosshair` | Render | Draws a local four-arm HUD crosshair. | `size`, `gap`, `thickness`, `outline`, `color`, `dynamic_movement`, `hide_vanilla` | Dynamic movement uses local horizontal velocity; hit detection and server packets are unaffected. |
| `better_nametags` | Render | Draws bounded local facts as stacked, color-coded billboard rows above nearby rendered players: name/friend, fraction-colored health, armor, then distance and held item. | `health`, `distance`, `armor`, `held_item`, `friend_status`, `range` | It processes only frustum-visible nearby players, skips the local player, and reads already-loaded health/equipment/friend facts. Zero armor and an empty hand produce no row. It does not modify vanilla name tags or expose hidden/unloaded players. |
| `no_weather` | Render | Hides local falling rain and/or snow precipitation, including during thunderstorms. | `rain`, `snow` | It is off by default and clears only the freshly extracted local `WeatherRenderState` precipitation columns at the tail of `WeatherEffectRenderer.extractRenderState`, so the following vanilla render draws nothing. It changes no weather packet, world state, biome, or lighting: server-driven sky darkening and separate lightning-bolt entities are unaffected. Disabling the module lets the next extraction repopulate columns and restores vanilla rendering. Acceptance: with the module enabled and `rain`/`snow` on, the matching precipitation stops rendering locally and returns on disable; the toggles are independent. Tests: `NoWeatherTest`. |
| `name_protect` | Render | Replaces the local profile name with a configurable alias in text reaching Minecraft's local font renderer. | `alias`, `case_sensitive`, `whole_name_only`, `maximum_replacements` | It is off by default and changes only rendered/locally measured text; the profile, signed chat, packets, logs, clipboard content, commands, and server-visible name remain unchanged. The style-preserving adapter processes at most 4,096 UTF-16 characters and 64 matches per text value. The default case-insensitive whole-name policy covers ordinary chat, tab/HUD text, tooltips, and world text that reaches the verified 26.2 `Font` preparation path; text baked into textures/images, content drawn by a custom renderer, or a name already truncated into separate fragments is outside that path. An alias containing the real name falls back to bounded asterisks. Disable restores vanilla rendering immediately. Covered by `NameProtectTest`; font method descriptors were verified against the 26.2 mapped client jar. |
| `saturation_display` | Render | Displays the local player's observed hunger saturation as a HUD readout. | none | It reads only Minecraft's local `FoodData` value, rounds for display, and never changes hunger, saturation, food consumption, or server state. Its enabled state and placement are local HUD-editor choices. |
| `entity_esp` | Render | Locally highlights selected rendered entities in Outline, Box, Glow, or Shader mode. | `players`, `hostiles`, `passive`, `items`, `projectiles`, `friends`, `range`, `maximum_entities`, `mode`, `line_width`, `color`, `friend_color`, `fill_color` | It selects at most 512 targets per frame/tick and classifies every non-hostile living entity as passive. Outline draws unfilled local Gizmo wireframes and Box adds the fill color. Glow and Shader route the same selection through Minecraft's genuine entity-outline post-processing pass by answering `Minecraft.shouldEntityAppearGlowing` for Helikon's tick-snapshotted targets only: Glow keeps the vanilla team-derived outline color, while Shader overrides the extracted `EntityRenderState.outlineColor` with the module/friend color (forced opaque, because the vanilla pass treats color 0 as no outline). No mode calls `Entity.setGlowingTag` or mutates server-provided entity flags; disable, mode changes away from Glow/Shader, and world changes clear the local target snapshot, which restores vanilla glow behavior exactly. Glow/Shader use vanilla's outline rendering rather than per-entity width, so `line_width` affects only Outline/Box and `fill_color` affects only Box. |
| `chams` | Render | Renders selected entities with Minecraft's occlusion-visible outline material so they show through walls. | `players`, `hostiles`, `passive`, `friends`, `range`, `maximum_entities`, `health_color`, `color`, `friend_color` | It is disabled by default and player-focused by default. Each tick it snapshots at most `maximum_entities` (capped at 512) locally rendered targets inside `range`, then forces only those entity IDs through Minecraft's native entity-outline post-processing pass by answering `Minecraft.shouldEntityAppearGlowing` for the snapshot and overriding the extracted `EntityRenderState.outlineColor` with the resolved color (forced opaque, because the outline pass treats color 0 as no outline). This produces a colored silhouette that remains visible through walls; it is an outline-based material and does not fill the model body. `health_color` colors living targets from green (full) to red (low); friends keep `friend_color`. It calls no `Entity.setGlowingTag` and mutates no server-provided entity flags, so disable and world changes clear the snapshot and restore vanilla glow behavior exactly. The server remains authoritative over entity presence and visibility; this is purely local rendering. |
| `base_finder` | Render | Incrementally scans loaded nearby blocks and highlights clusters of configurable player-built-base evidence. | `evidence_blocks`, `minimum_evidence`, `cluster_radius`, `horizontal_range`, `vertical_range`, `scan_budget`, `maximum_markers`, `tracers`, `line_width`, `color`, `fill_color` | It checks 64–2048 already-loaded blocks per client tick, caches at most 512 matching blocks, and renders at most 512 markers. The default three-block cluster heuristic reduces isolated false positives but cannot prove that a structure is player-built; villages and other generated structures may contain the same blocks. It never requests or reveals unloaded chunks, and a complete scan pass can take several seconds at larger ranges. Disable and world changes clear all transient results. Covered by `BaseFinderTest` plus the existing bounded scanner tests. |
| `block_esp` | Render | Incrementally scans a bounded local cube for configured block IDs and draws Gizmo boxes. | `blocks`, `block_colors`, `horizontal_range`, `vertical_range`, `scan_budget`, `tracers`, `line_width`, `color`, `fill_color` | It checks 64–2048 blocks per client tick and retains at most 512 matches. `block_colors` accepts bounded `block=#RRGGBB` or `block=#AARRGGBB` entries; malformed IDs/colors fall back to the shared local color. |
| `open_water_esp` | Render | Highlights loaded water surfaces whose four 5×5 layers satisfy Minecraft's open-water fishing shape. | `horizontal_range`, `vertical_search`, `scan_budget`, `maximum_sites`, `always_on_top`, `line_width`, `color`, `fill_color` | Disabled by default. It incrementally checks 1–32 loaded columns per tick, searches only the configured local vertical band, and retains the nearest 16–512 sites. A site requires uniform source-water layers with empty collision followed only by uniform air/lily-pad layers, matching the verified 26.2 `FishingHook` rule. Results can take a complete scan pass to appear or update, do not predict cast reach or line of sight, request no chunks, and cannot guarantee the server will accept a cast or award treasure. Disable and world change clear the local cache; `OpenWaterEspTest` covers layer ordering, mixed/blocked layers, bounds, defaults, and cleanup. |
| `cave_finder` | Render | Incrementally samples loaded terrain below the player and highlights walkable underground air pockets. | `minimum_depth`, `scan_depth`, `horizontal_range`, `scan_budget`, `sample_spacing`, `minimum_open_sides`, `require_floor`, `require_buried`, `maximum_markers`, `tracers`, `line_width`, `color`, `fill_color` | It checks 64–2048 already-loaded positions per client tick, caches at most 512 matches, and renders at most 512 through-wall markers. The default world-grid spacing, two-block headroom, floor, horizontal-opening, depth, and heightmap tests reduce noise but are only a cave heuristic: underground rooms can match and narrow or water-filled caves can be missed. It never requests or reveals unloaded chunks. Disable and world changes clear all transient results. Covered by `CaveFinderTest` and the existing bounded scanner tests. |
| `mob_spawn_esp` | Render | Incrementally scans loaded nearby floors and marks positions that approximate ordinary hostile-mob spawn conditions. | `horizontal_range`, `vertical_range`, `scan_budget`, `sample_spacing`, `maximum_markers`, `maximum_block_light`, `maximum_sky_light`, `respect_player_distance`, `always_on_top`, `line_width`, `color`, `fill_color` | It checks 64–2048 already-loaded positions per client tick, caches at most 512 matches, and renders at most 512 markers. A marker requires two blocks of air, a floor accepted by Minecraft 26.2 for a representative ordinary hostile (`EntityTypes.ZOMBIE`), non-Peaceful difficulty, and light values within the configured thresholds. The optional distance gate models the vanilla 24-block player exclusion. This remains a planning approximation: the client cannot predict server mob caps, spawn-cycle randomness, biome/entity-specific placement, world-border rules, or transient entity collisions, and the server is authoritative. Grid sampling can skip positions; updates can take one full scan pass. It never requests or reveals unloaded chunks, and disable/world changes clear transient results. Covered by `MobSpawnEspTest` and the existing bounded scanner tests. |
| `new_chunks` | Render | Marks chunk coordinates first seen or reloaded by the local chunk cache during the current enabled session. | `mark_first_seen`, `mark_reloads`, `retain_unloaded`, `lifetime_seconds`, `maximum_chunks`, `range_chunks`, `vertical_span`, `always_on_top`, `line_width`, `fill_opacity`, `first_seen_color`, `reload_color` | Disabled by default. “First seen” means the first `FULL` cache load observed while enabled, not proof that the server freshly generated the chunk; “reloaded” requires an observed unload followed by another load. It reads no unloaded data and sends no chunk requests. Markers expire, rendering is range/cap bounded, the first-seen history is capped at 4,096 coordinates, and disable or world change clears all session state. |
| `tracers` | Render | Draws local eye-to-entity lines for selected nearby rendered entities. | `players`, `hostiles`, `passive`, `items`, `projectiles`, `friends`, `range`, `maximum_entities`, `line_width`, `color`, `friend_color` | It renders at most 512 local lines per frame. Lines are a visual aid only and do not imply visibility, targeting, or reach through blocks. |
| `player_finder` | Render | Marks distant locally loaded players with configurable world-space tracers, vertical beacons, and name/distance labels. | `distance`, `include_friends`, `include_spectators`, `tracers`, `beacons`, `labels`, `beacon_height`, `always_on_top`, `maximum_players`, `line_width`, `color`, `friend_color` | Disabled by default; friends and spectators are excluded by default. It sorts nearest-first and draws at most 64 results from `ClientLevel.players()` per frame. It sends no requests and cannot reveal unloaded, untracked, vanished, other-dimension, or otherwise absent players; the maximum distance is only a filter over what the server has already synchronized. `PlayerFinderTest` covers identity/defaults, friend/spectator and distance filtering, ordering, caps, labels, and setting bounds. |
| `breadcrumbs` | Render | Draws a bounded local trail of recent player positions. | `maximum_points`, `maximum_age_seconds`, `sampling_distance`, `always_on_top`, `line_width`, `color` | The trail is session-only, clears on disable or world change, and has no persistent export. |
| `trajectories` | Render | Predicts local paths for visible in-flight projectile entities. | `arrows`, `tridents`, `snowballs`, `eggs`, `ender_pearls`, `splash_potions`, `maximum_steps`, `maximum_projectiles`, `line_width`, `color`, `impact_color` | It simulates only currently frustum-visible in-flight projectiles, uses verified family physics/tick ordering, and stops at the first local block collision. It does not alter projectile state, predict entity collisions, or provide a held-item aiming preview. |
| `projectile_warning` | Render | Highlights incoming projectiles on a short-horizon collision course with the local player. | `arrows`, `tridents`, `snowballs`, `eggs`, `ender_pearls`, `splash_potions`, `exclude_friend_projectiles`, `show_label`, `hit_radius`, `warning_seconds`, `range`, `maximum_projectiles`, `line_width`, `color`, `fill_color` | It is off by default. It draws a warning box (and optional seconds-to-impact label) only for frustum-visible projectiles whose straight-line path at the current per-tick velocity passes within `hit_radius` of the local player within `warning_seconds`. It ignores stationary/stuck projectiles, projectiles moving away, the local player's own projectiles, and (by default) friend-owned projectiles; friends are resolved through `FriendManager`. The threat check is a conservative linear estimate that intentionally ignores gravity and drag, so it complements rather than replaces `trajectories`. Because it is a world overlay it warns about projectiles in the current view, not ones behind the camera; it never changes projectile state, velocity, or packets and the server stays authoritative over damage. |
| `projectilepreview` | Render | Predicts where a currently held projectile would travel before it is fired or thrown, from the player's current aim. | `bows`, `crossbows`, `tridents`, `throwables`, `offhand`, `maximum_steps`, `line_width`, `color`, `impact_color` | Defaults off. It previews a bow only while it is actually being drawn (path lengthens with draw power and hides below the vanilla minimum firing charge), a crossbow only while loaded, and tridents/throwables while held; the offhand is limited to throwables. It reuses verified per-family launch geometry (`shootFromRotation`) and physics and stops at the first local block collision. The path launches from the eye position and ignores per-shot inaccuracy, entity collisions, wind charges, and riptide; the server stays authoritative, so an actual shot may diverge, be rejected, or rubber-band. |
| `remote_view` | Render | Switches the local render camera to one eligible, already-loaded entity chosen by crosshair, nearest distance, or exact player name. | `target_mode` (`crosshair`, `nearest`, `named_player`), `entity_types`, `target_name`, `range`, `include_friends`, `include_invisible`, `include_spectators`, `retarget_on_loss` | Disabled by default; eligible types default to players, while friends, invisible entities, and spectators are excluded. It examines at most 1,024 locally rendered entities, never loads or requests an entity, and cannot view unloaded, vanished, or other-dimension targets. It calls only Minecraft's local `setCameraEntity`: it does not move/rotate the target, move the player, suppress ordinary user input, or construct packets. Disable, world loss, target loss, and selection-setting changes restore the valid camera that preceded RemoteView (or the local player as fallback). Without `retarget_on_loss`, losing a target leaves the module enabled but safely back on the restored camera until it is toggled or its selection changes. `RemoteViewTest` covers modes, filtering, deterministic selection, selection revisions, defaults, and cleanup. |
| `true_sight` | Render | Draws local translucent boxes around selected invisible entities. | `players`, `hostiles`, `passive`, `range`, `maximum_entities`, `transparency`, `line_width`, `color` | It draws a box only when the entity is invisible to the local player and frustum-visible, without forcing vanilla invisible models to render. This safe initial visualization supports living categories only and does not change entity state. |
| `radar` | Render | Draws a compact local overhead entity radar HUD with an optional terrain minimap. | `shape` (`circle`, `square`), `minimap`, `rotate`, `zoom`, `players`, `hostiles`, `passive`, `items`, `friends`, `maximum_entities`, `background_color`, `entity_color`, `friend_color` | Its enabled state and placement are local HUD-editor choices. It reads only currently loaded entities and surface blocks, caps entity points at 512, and never reveals unloaded data. The cached minimap terrain stays underneath the existing entity dots. |
| `arrows` | Render | Draws local directional arrows around the screen center pointing toward enemies outside the field of view. | `players`, `hostiles`, `passive`, `friends`, `range`, `field_of_view`, `ring_radius`, `arrow_length`, `arrow_width`, `maximum_targets`, `color`, `friend_color` | Disabled by default; friends are excluded unless `friends` is on. It reads only currently loaded entities within range, marks a target only when it lies outside the configured angular cone (an approximation of the on-screen area, not pixel-exact frustum clipping), draws the nearest targets first up to `maximum_targets`, and hides with the rest of the custom HUD on panic. It reveals no unloaded entities and changes no game or server state. |
| `xray` | Render | Locally rebuilds chunk geometry to retain only configured block models. | `blocks`, `opacity` | The list accepts at most 32 valid semicolon-delimited block IDs; if none are valid, the module keeps normal geometry rather than hiding every block. Enable/disable and setting changes rebuild loaded local geometry; it does not request chunks or alter world data. Some block-model/material edge cases may need port-specific validation after Minecraft updates. |
| `storage_esp` | Render | Incrementally scans bounded loaded chunks for selected block entities and draws local boxes. | `chests`, `barrels`, `shulkers`, `furnaces`, `hoppers`, `spawners`, `custom_block_entities`, `horizontal_range`, `vertical_range`, `scan_budget`, `line_width`, `color`, `fill_color` | It checks 64–2048 blocks per client tick, retains at most 512 matches, and requires both a selected ID and a local block entity. Boxes are frustum-culled; newly loaded or changed storage can take one scan pass to appear or disappear. |
| `mini_player` | Render | Draws a transparent local HUD miniature of the current player model. | `rotation`, `scale`, `armor` | Its enabled state and placement are local HUD-editor choices. It intentionally draws no panel background or outline. Armor off clears only the temporary extracted render state; player inventory/equipment is never changed. |
| `damage_indicators` | Render | Draws fading local damage labels for observed nearby living entities. | `players`, `hostiles`, `passive`, `range`, `maximum_tracked_entities`, `maximum_indicators`, `lifetime_seconds`, `rise_distance`, `color` | It reports only a locally observed positive health delta while Minecraft reports a current hurt state, so some damage may not have an amount available. Nearest observations and labels are independently bounded, frustum-culled, and never affect combat. |
| `explosions` | Render | Draws a local wireframe sphere at the expected damage radius of nearby primed explosion sources. | `tnt`, `creepers`, `tnt_minecarts`, `end_crystals`, `armed_creepers_only`, `segments`, `maximum_sources`, `line_width`, `color`, `show_radius_label` | It reads only already-loaded local entities (primed TNT, primed TNT minecarts, end crystals, and creepers) and draws at most 128 frustum-visible sources per frame. The sphere is the exact vanilla entity-damage radius (`power * 2.0`): 8 for TNT/TNT minecart, 6 for a creeper, 12 for a charged creeper or end crystal. Actual block destruction is ray-traced and terrain dependent, so it may fall short of the sphere; the module is an awareness aid and never changes entity state or sends packets. `armed_creepers_only` (default on) limits creepers to those actively swelling or ignited. |

## Combat modules

| ID | Category | Description | Settings | Limitation |
| --- | --- | --- | --- | --- |
| `anti_bot` | Combat | Locally marks player entities matching conservative bot heuristics. | `tab_list_presence`, `impossible_state`, `minimum_spawn_age`, `duplicate_names`, `invisible`, `missing_profile` | It reads only already-loaded player/tab-list facts. A heuristic is not proof of a bot, is never sent anywhere, and is used only as a local exclusion input. |
| `trigger_bot` | Combat | Requests one normal attack only for the current eligible crosshair target. | `delay_ticks`, `weapon_required`, `players`, `hostiles`, `passive`, `exclude_friends` | It requires locally observed line of sight, normal attack cooldown, and optionally a conventional melee-item ID. It never changes reach or creates an attack packet. |
| `bow_aim_assist` | Combat | Smooths local aim toward one eligible target while Use is held with a bow. | `players`, `hostiles`, `passive`, `exclude_friends`, `range`, `field_of_view`, `prediction`, `projectile_speed`, `gravity`, `adjustment_speed` | It adjusts only local view rotation at a bounded rate, draws one local Gizmo outline, and never draws/releases a bow, sends a custom rotation packet, or targets through blocks. Prediction is an estimate from visible local velocity. |
| `arrow_dmg` | Combat | Attempts to increase a bow arrow's launch velocity with a bounded movement sequence immediately before release. | `strength`, `minimum_draw_ticks`, `stationary_packets`, `sprint_signal` | Defaults off and acts only on an actually drawn bow released through Minecraft's normal held-item path. It sends at most one vanilla sprint command plus six well-formed vanilla position updates (four stationary, one bounded backward offset, and one return) before the ordinary release packet; it never sends malformed data or loops without a user release. Arrow damage and movement remain server-authoritative: servers may ignore the effect, cap the velocity, correct/rubber-band the player, or disconnect them, especially at high `strength`. Movement-buffering modules can also break the required ordering. |
| `aim_assist` | Combat | Nudges the local crosshair toward one eligible melee-range target, independent of a held bow. | `players`, `hostiles`, `passive`, `exclude_friends`, `range`, `field_of_view`, `rotation_speed`, `priority` (`distance`, `health`, `angle`), `require_weapon`, `require_attack_key` | It adjusts only local view rotation at a bounded rate and requires local line of sight, so it does not aim through solid blocks. It sends no custom rotation packet; the server stays authoritative and may reject, correct, or rubber-band any resulting attack. Optional gates limit it to a held melee weapon and/or the Attack key. |
| `critical_assist` | Combat | Times a normal held attack during an ordinary falling critical opportunity. | `delay_ticks`, `exclude_friends` | It requires the user to hold Attack, a normal cooldown, downward fall, no ground/water/climb/fall-flying state, and local line of sight. It never moves the player or uses packet exploits. |
| `mace_dmg` | Combat | Times an ordinary mace attack after a real fall has accumulated the configured smash-damage opportunity. | `minimum_fall_distance`, `minimum_attack_charge`, `require_attack_key`, `exclude_friends`, `delay_ticks` | Defaults off. It attacks only the eligible non-friend entity under the crosshair while descending with a mace, outside water, climbing, fall-flying, passenger, and grounded states. It does not create height, edit fall distance, teleport, or send fabricated movement packets; the configured 1.6-50 block threshold is based only on locally observed real fall distance. Minecraft 26.2 requires more than 1.5 blocks for a smash and computes the bonus from server-observed fall distance, so the server may still reject the ordinary attack, see a different distance, or award less damage. |
| `jump_reset` | Combat | Times a normal Jump input when the local player is hit to reduce received knockback. | `require_movement`, `cooldown_ticks` | It requests only Minecraft's ordinary local Jump on the tick a fresh hurt-time rising edge is observed while grounded and no screen is open. `require_movement` (on by default) additionally needs local horizontal velocity above a small threshold, so idle fall damage does not trigger it. Detection uses only the local hurt animation, so any damage source can qualify. It sends no custom packet; the server stays authoritative and may still apply full knockback, correct, or reject the jump. |
| `auto_respawn` | Combat | Requests one normal vanilla respawn after local death. | `delay_ticks` | It calls the same local-player respawn path as Minecraft's Death Screen and sends at most one ordinary respawn request for each observed death. The server remains authoritative and can delay, reject, or otherwise handle the request. |
| `auto_potion` | Combat | Selects and normally uses a local restorative hotbar potion at low health. | `health_threshold`, `potion_whitelist`, `kind` (`splash`, `drink`, `both`), `delay_ticks` | It accepts only hotbar potions whose observed effects include Instant Health and whose potion ID is whitelisted. It uses Minecraft's ordinary held-item path, never throws/drinks a non-restorative potion, and restores only a slot it still owns. |
| `auto_leave` | Combat | Disconnects from multiplayer at configured locally observed danger thresholds. | `low_health`, `health_threshold`, `falling`, `fall_threshold` | It leaves only a non-local multiplayer connection through Minecraft's ordinary disconnect flow after low health (including absorption) or fall distance reaches an enabled threshold. | `auto_soup` | Combat | Uses player-owned mushroom stew at a low local health threshold. | `health_threshold`, `delay_ticks` | It considers only hotbar mushroom stew, makes one normal main-hand use request, and restores a prior selected slot only while it still owns it. | `click_aura` | Combat | Attacks the nearest locally observed eligible target while Attack is held. | `players`, `hostiles`, `passive`, `exclude_friends`, `range`, `delay_ticks` | It requires the normal vanilla cooldown, local line of sight, a held Attack key, and targets no farther than 6 blocks. ## ClickGUI

Modules appear in the ClickGUI (Right Shift) under their `ModuleCategory`. The
**Active** row above the categories lists enabled modules from every category;
its row checkbox turns a utility off without leaving the view.
Selecting a module shows its name, category, ID, description, an enabled
toggle, and controls for its `BooleanSetting`, `NumberSetting`, `ColorSetting`,
and finite `EnumSetting` values. Colors use strict `#AARRGGBB` tokens; click
an enum row to cycle its documented choices. Integer, string-list, selector,
multi-select enum, range, regex, and keybind settings use compact validated
text fields. Lists/selectors use `;`, multi-enums use `,`, ranges use
`minimum..maximum`, and keybind values use
`keyboard|mouse:code:toggle|hold|press_once[:shift+control+alt+super]`.
Invalid text is red and leaves the last valid stored value unchanged.
Toggles are dispatched through `ModuleRegistry`, so a module that throws
during `onEnable`/`onDisable` is isolated and force-disabled instead of
crashing the client. Setting edits and enabled state persist to
`config/helikon/global.json` when the screen closes.

The settings panel also has **Reset module** plus small **R** buttons for each
visible setting. Reset operations use the existing validated setting defaults.
When a module has more controls than fit in the panel, hover the right-hand
panel and use the mouse wheel; controls, editable fields, and the scrollbar
move together inside the clipped viewport.
Its **Bind** row captures a keyboard key or mouse button locally with any held
Shift, Control, Alt, or Super modifiers; Escape cancels capture,
Backspace/Delete clears the bind, and the key that opens the Helikon GUI is
rejected. Existing activation mode (`toggle`, `hold`, or `press_once`) is
preserved when rebinding, and a local warning identifies exact module-bind
collisions. Drag an unused portion of the header to move the
window, or its bottom-right handle to resize it. The clamped top-left position
and custom dimensions persist in `global.json`.

Select **Theme** from the ClickGUI header to open its local palette selector.
The initial themes are Midnight, High Contrast, and Ocean; selecting one
changes the panel immediately and persists it locally. Color settings also
show an in-panel ARGB picker below their validated text value; drag any of the
four channel tracks to update that one local color channel.

When no text field is focused, the ClickGUI supports keyboard navigation:
Left/Right changes the category (and exits search), Up/Down moves the selected
module row with wrapping and automatic list scrolling, and Enter or Space
toggles the selected module through `ModuleRegistry`.

## Commands and keybinds

`.taco` prints a harmless taco easter egg only in local chat. Local command-name Tab completion is built into the chat screen for every registered dot command.

Modules can also be controlled through local chat commands (`.toggle`,
`.setting`, `.reset`, `.bind`, `.unbind`, `.profile` — see the README) and
per-module keybinds with `toggle`, `hold`, or `press_once` activation. Profile
commands save/load/duplicate/rename the complete local module and ClickGUI
snapshot without transmitting it. The default profile is applied at startup,
and a matching local server/world association overrides it on join. Import/export uses Helikon's local
`imports/` and `exports/` folders rather than arbitrary paths. A profile can
also be marked as the persisted default. Keybinds never fire while a screen is
open. Server and singleplayer associations are also local profile preferences;
they neither inspect nor send server data beyond the user-provided identifier.

## Friends

Friends are local player-name entries, managed through `.friend list`,
`.friend add`, `.friend remove`, and `.friend color`. Enable
**OneClickFriends**, then middle-click a targeted player in the game world to
add or remove that name without opening chat.
Each entry stores a local ARGB render color in `friends.json`; no friend data is
sent to a server. Targeting modules will use the friend list for their default
friend-exclusion policy when those modules are introduced.

## Waypoints

`.waypoint add <name>` saves the current block position; supplying `x y z`
saves manual coordinates in the currently loaded server/world and dimension.
`.waypoint list` orders enabled local entries by distance and reports a compass
direction. Waypoints can be removed, renamed, toggled, recolored, or given a
small optional icon token with local commands. The minimal Waypoints HUD shows
up to three nearest enabled entries with distance and direction, and hides
entries from any other server/world or dimension.

Death and logout coordinates are now separate enabled-only session snapshots;
they deliberately do not create automatic waypoints. The first HUD indicator
has no separate HUD-editor position or world-space beacon; those rendering and
layout controls remain future work.

## Macros

Macros are configured through `.macro`. Create a global or currently connected
server-scoped macro, then append explicit `local`, `chat`, `command`, or
`delay` actions. A local action must start with `.` and stays inside the local
dispatcher; chat text cannot start with `.` or `/`; Minecraft command text
omits `/`; delay actions are bounded to 1-12,000 ticks. Run one macro at a time
with `.macro run <name>` and stop it with `.macro stop`. The runner pauses while
a screen is open, executes no more than one action per client tick, and stops a
server-scoped macro if the current server changes.

Macros are not a scripting engine: they cannot execute code, read files, or
download content. They do not bypass server authority, rate limits, or normal
chat/command packet formats.

## Panic

`.panic` disables every enabled module through `ModuleRegistry`, so current and
future modules run their own `onDisable` restoration (for example brightness,
timer, FOV, or step changes). It closes any Helikon GUI screen, stops queued
macro work, and hides all custom HUD for the current session without deleting
or changing user configuration. Use `.panic restorehud` to show HUD again
without re-enabling modules. `.panic bind <key>` stores a separate local
keyboard/mouse panic bind; chat and ordinary screens suppress it, while Helikon
screens permit it as an emergency close control.
